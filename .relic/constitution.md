## Core Principles

### I. TypeScript Strict Mode — No Escape Hatches
`strict: true`, `noUncheckedIndexedAccess: true`, `noImplicitOverride: true`, and `noFallthroughCasesInSwitch: true` are enforced project-wide. These prevent the most common categories of runtime errors. `noUnusedLocals` and `noUnusedParameters` are intentionally relaxed to reduce friction during development.
Source: `tsconfig.json` (explicit)

### II. AI Workflow Commands Are Prompts, Not TypeScript
`specify`, `clarify`, `plan`, `analyse`, `tasks`, `implement`, and `fix` are AI prompt files in `templates/prompts/`. Their TypeScript stubs in `packages/core/src/commands/` exist only to test context assembly — they are not the implementation. The business logic of these commands must remain in the prompt templates. Do not migrate workflow behaviour into TypeScript.
Source: `docs/implementation.md` Phase 4 and Phase 8 (explicit)

### III. Zero Runtime Dependencies — Everything Bundled Inline
The npm package has no runtime `dependencies`. `@relic/core`, `commander`, and all templates are inlined by the bundler at build time. The binary ships no loose files. Template content is accessed only through the `TEMPLATES` map produced by `scripts/embed-templates.ts`. Never add a `dependency` to `packages/cli-node/package.json` — add it as a `devDependency` and let the bundler inline it.
Source: `docs/distribution.md`, `packages/cli-node/package.json`, `scripts/embed-templates.ts` (explicit)

### IV. No Bash, No jq, No sed in User Projects
All CLI utilities used by AI agents are native TypeScript commands (`relic context`, `relic scaffold`, `relic validate`). This is non-negotiable for Windows portability and minimal CI environments. When adding a utility operation that an AI agent calls, extend the TypeScript CLI — never add a bash script dependency to the user's project.
Source: `docs/implementation.md` Phase 8 rationale (explicit)

### V. All Utility Commands Output JSON by Default
`relic context`, `relic validate`, `relic scaffold`, `relic search`, and `relic deep-search` all output machine-readable JSON by default. Human-readable output requires the `--text` flag. This ensures AI agents can parse output programmatically without text parsing. New utility commands must follow this convention.
Source: Consistent pattern across `commands/*.ts` (confirmed)

### VI. Template Embedding Is the Build Contract
All template content (`.md` prompt files, engine instruction files, spec scaffolding files) is baked into `packages/core/src/generated/templates.ts` at build time. Running `bun run build:templates` must always be the first step in any build pipeline — it is a prerequisite for every other build target. Accessing template content outside of the `TEMPLATES` map is forbidden.
Source: `package.json` scripts, `scripts/embed-templates.ts` (explicit)

## Tech Stack

**Language & Runtime:**
TypeScript 6.x with full strict mode. Bun for development, building, and scripts. Node.js 18+ for the published npm bundle. ESM-only (`"type": "module"`, `verbatimModuleSyntax: true`).

**Framework(s):**
`commander` for CLI argument parsing (inlined in the bundle — no runtime dependency). No web framework.

**Data:**
Markdown + JSON files on disk. No database. File I/O via `packages/core/src/utils/fs.ts` — use these utilities, not raw Node.js `fs` calls, for testability.

**Required patterns:**
Monorepo with `packages/core` (all business logic), `packages/cli-node` (npm bin), `packages/cli-python` (PyPI wheel). Business logic lives only in `@relic/core`. CLI packages are thin entry points that delegate to `@relic/core`.

**Forbidden patterns:**
No `any` type without a comment justifying it. No bash scripts in user-facing output. No loose files in the published binary. No runtime `dependencies` in `packages/cli-node/package.json`. No workflow business logic in TypeScript (see Principle II).

## Architecture

**Structure:**
Bun monorepo. `packages/core` — all TypeScript business logic, exported as `@relic/core`. `packages/cli-node` — thin bin wrapper, builds to Node.js bundle for npm. `packages/cli-python` — Python shim that embeds a platform-compiled Bun binary for PyPI.

**Key decisions:**
- AI workflow commands (`specify`, `plan`, etc.) intentionally live in prompt files, not TypeScript code
- Templates embedded at build time via `scripts/embed-templates.ts` — no runtime path resolution
- Bash scripts were deliberately removed (Phase 8) — all CLI utilities are TypeScript
- Production binary (`bin.ts`) exposes only user-facing and AI-utility commands; workflow stubs are debug-only
- `.relic/current-spec` is gitignored — each team member tracks their own active spec session

**Layer boundaries:**
`packages/cli-node` and `packages/cli-python` must not contain business logic. All logic lives in `packages/core/src/`. CLI packages import from `@relic/core` only.

## Development Workflow

**Branching:**
`NNN-slug` branches enable automatic spec inference (e.g. `001-auth` → spec `001-auth`). Use this convention when working on a spec to get zero-config spec resolution.

**Commits:**
Conventional commits: `feat`, `fix`, `chore`, `docs`, `refactor`. Version bump commits use `chore: bump version to X.Y.Z` (enforced by `scripts/publish.ts`).

**Release:**
`bun run publish <version>` is the single release entry point. It bumps all 6 version files, creates a `release/vX.Y.Z` branch, commits, tags, and pushes. Never manually edit version fields across files — use the publish script.

**Review:**
No formal review policy documented. CI must pass before merging.
